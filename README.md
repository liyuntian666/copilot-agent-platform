# Copilot 智能操作平台

> 面向企业业务场景的 AI Agent 执行平台
> 支持任务拆解、工具调用、参数提取、结果摘要、人类监督与多任务协同执行

---

## 项目简介

Copilot 智能操作平台是一个基于 Python 构建的多工具协同执行系统，主要用于解决企业内部 ERP / 业务系统中大量需要人工跨系统操作的问题。

系统通过大模型完成：

* 任务理解
* 工具选择
* 参数提取
* 参数补全
* 结果摘要
* 多任务规划

同时结合工程化机制，解决：

* 工具调用不稳定
* 上下文过长
* 多轮任务循环
* 人类监督
* 提示词注入
* 长链路可视化

等实际业务问题。

# 核心设计

---

## 1. 异步任务执行

用户请求进入 `mesh_query()` 后，不直接同步执行，而是提交到线程池异步处理。

这样做的目的是：

* 提高接口响应速度
* 避免长任务阻塞
* 支持多任务并发执行

```python
executor.submit(run_task, task)
```

---

## 2. 任务状态可视化

由于：

* 外部 API 调用耗时
* 多次大模型推理耗时

所以不能让用户只看到：

```text
正在处理中...
```

系统会把每一步执行状态写入 MongoDB 中的 `Task` 类，用于前端实时展示：

* 当前执行阶段
* 调用了哪些 API
* 当前等待什么操作
* 哪一步需要人工确认

---

## 3. 工具选择优化

### 问题

如果把所有工具描述直接放入 Prompt：

* token 消耗高
* 干扰信息多
* 工具误选率高

### 优化

我采用了：

```text
工具摘要
+
工具分类
+
查询改写
+
重排序
```

的方式优化工具选择。

---

## 4. 查询改写

用户问题通常是口语化的。

例如：

```text
帮我看看北京能发什么物流
```

并不适合直接检索工具。

因此会先进行：

```text
Query Rewrite
```

把自然语言转换成更适合工具召回的表达。

---

## 5. 工具初筛 + 重排序

实践中发现：

* topk 太小 → 工具漏召回
* topk 太大 → token 爆炸

最终测试效果较好的方案：

```text
初筛：
2 × topk = 10

重排序后：
topk = 5
```

案例：

某“配送区域查询供应商”工具：

```text
初筛排名：6
重排序后：1
```

---

## 6. 参数提取与校验

大模型提取参数后：

并不能默认可信。

因此我增加了：

* 必填项检查
* 参数类型检查
* 缺失参数列表生成

例如：

```json
{
  "missing_params": [
    "product_id",
    "supplier_id"
  ]
}
```

---

## 7. 参数补全

参考 RestGPT 的思路：

当参数缺失时：

不是直接失败。

而是：

```text
调用参数 API
→ 自动补全业务参数
```

提高任务完成率。

---

## 8. 外部系统默认不可靠

在工程上：

我默认：

```text
所有外部系统都不可靠
```

因此：

* 大模型调用
* API 调用

都增加了：

```text
重试机制
+
异常处理
+
退避机制
+
结果格式检查
```

例如：

```python
retry_count -= 1
time.sleep(random.randint(1, 20))
```

用于应对：

* 限流
* 网络波动
* 服务不稳定

---

## 9. 多模型 Prompt 模板

不同模型：

Prompt 表现差异明显。

例如：

某些模型在日期参数提取时：

容易输出错误格式。

因此我设计了：

```text
统一 Prompt 模板
+
模型差异部分抽离
```

构建场景化 Prompt 体系。

---

## 10. 长上下文处理

批量查询时：

上下文可能超过模型限制。

相比高成本摘要：

我采用了更工程化的策略：

```text
超过最大长度
→ 截断
```

例如：

```text
最大长度：30000

超过后：
仅保留核心部分
```

避免：

* token 爆炸
* 摘要耗时过长

---

## 11. Prompt Injection 防护

工具执行前：

增加提示词注入检测。

包括：

### 程序规则

```text
价格必须 > 0
```

### LLM 判断

```text
是否存在恶意 Prompt
```

降低危险调用风险。

# 多任务处理

---

## 为什么没有直接使用 ReAct？

实践中发现：

ReAct 在复杂业务场景下：

容易：

* 错误率高
* 重试次数多
* token 消耗高

---

## 为什么没有完全采用 Plan-and-Execute？

Plan-and-Execute：

虽然能先规划。

但：

```text
业务计划会动态变化
```

例如：

```text
库存充足
→ 需要查询物流
→ 没有物流商
→ 再调用物流 API
```

因此：

我参考了 RestGPT 的思路：

```text
动态规划
+
执行过程中持续修正
```

---

## 多任务循环检测

多任务中：

可能出现：

```text
重复调用同一个 API
```

因此增加：

```text
循环检测
```

规则：

```text
连续 3 次
相同 API
且结果相同
→ 判定循环
```

# 人类监督机制

对于：

* 删除
* 修改
* 下单

等高风险操作。

不能让模型直接执行。

因此增加：

```text
等待人类反馈
```

任务状态：

```text
未运行
正在运行
等待人类反馈
任务完成
任务中止
```

# Task 类设计

`Task` 不仅用于前端展示。

同时也是：

```text
任务恢复
+
上下文恢复
+
人工介入恢复
```

的核心数据结构。

```python
task_id          # 任务唯一ID
status           # 当前状态
task_type        # 任务类型
raw_query        # 用户原始请求
changed_query    # 执行中变化后的请求
curr_task_desc   # 当前任务描述
nodes            # 前端调用链节点
edges            # 前端调用链边
graph_title      # 调用链标题
system_output    # 输出结果
curr_tool_id     # 当前待确认工具
```

# 人类反馈意图识别

用户反馈可能是：

* 同意
* 拒绝
* 修改
* 补全参数
* 更换工具

因此：

不能只靠简单关键词。

---

## 分层意图识别

采用：

```text
关键词 / 正则
↓
轻量模型
↓
大模型
```

逐层判断。

这样做的好处：

* 降低 token 消耗
* 提高响应速度
* 降低误判率

# 我对 AI Agent 的理解

我认为 AI Agent 的核心能力包括：

## 1. 根据经验调整行为

Agent 是否能：

* 根据历史反馈调整行为
* 越来越少犯错

是很关键的能力。

---

## 2. 工具使用能力

真正困难的：

不是“会调用工具”。

而是：

```text
如何稳定调用工具
如何处理异常
如何恢复执行
```

---

## 3. 规划能力

复杂任务中：

计划会不断变化。

因此：

规划能力不仅是：

```text
生成计划
```

更重要的是：

```text
动态修正计划
```

# 项目思考

这个项目让我真正意识到：

> AI Agent 最大的问题不是“能不能调用工具”，
> 而是“能不能在真实业务里稳定、可控、可恢复地执行任务”。

很多 Demo 看起来很强。

但真正进入工程环境后：

会遇到：

* token 成本
* API 不稳定
* 上下文爆炸
* 工具误选
* 参数缺失
* 循环调用
* 人类监督
* 安全风险

这些问题。

而这些问题：

往往才是真正的 Agent 工程化难点。
